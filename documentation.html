Section 01

Problem Statement & Objective

GitLab's Handbook and Direction pages contain thousands of pages of knowledge about processes, strategy, teams, engineering practices, and company values. Employees and candidates who need quick answers currently have to manually search or browse these documents.

The goal was to build a conversational AI assistant that answers questions grounded in this content — not hallucinated general knowledge — and presents it in a clean, production-quality interface.

Core Requirements

  • Chatbot interface over GitLab Handbook and Direction content
  • Streaming responses with follow-up question suggestions
  • Source citations linking back to handbook pages
  • Error handling and off-scope request guardrailing
  • Public deployment accessible via a live URL
  • Detailed setup documentation and GitHub repository
Section 02

System Architecture

The application is built entirely on Next.js 16 (App Router) and deployed on Vercel — a single platform for both frontend and backend. This eliminates CORS configuration, removes the need to manage two separate services, and provides zero-config CI/CD on every git push.

RAG PIPELINE — lib/orchestrator.js
1. checkGuardrails() → Block harmful / off-scope queries before any LLM call
2. routeQuery() → Decide: clarify / reject / vector / hybrid
3. embedText() → Gemini embedding-001 (3072-dim), query enrichment for short replies
4. searchVector() → Cosine similarity over local JSON index, URL-diverse top-K
5. generateAnswer() → Gemini 2.0 Flash with full multi-turn history
6. evaluateCritic() → Lexical grounding check — automatic retry if score < 8%
7. generateFollowups()→ 3 contextual follow-up questions (Groq fallback)
🔐

Auth & Storage

JWT (HS256) + bcryptjs. Conversations persisted in Vercel KV (Upstash Redis) with in-memory fallback for local dev.

⚡

Streaming (SSE)

Word-by-word server-sent events via ReadableStream. Named events: message, meta, sources, followups, done, error.

🗂️

Knowledge Index

44MB vector_index.json committed to repo, module-level cached on Vercel. No external vector DB required.

🎨

UI/UX

Light/dark theme with flash-free system detection. Full GFM markdown rendering via react-markdown + remark-gfm.

Section 03

Technical Design Decisions

Hybrid Retrieval: Vector + Knowledge Graph

Plain vector search returns semantically similar chunks but can miss relational questions. A knowledge graph of entities extracted from the handbook allows graph-based retrieval for relational queries. The query router selects the mode:

RouteTrigger ConditionRetrieval Strategy
vectorDefault semantic queryCosine similarity over all indexed chunks
hybridRelational terms: connect, between, impact, owner…Vector + knowledge graph entity lookup
clarify≤2 words or vague pattern (tell me more, go on…)Ask user for more detail — no retrieval
rejectExplicit off-topic terms (weather, recipe, crypto…)Polite refusal — no LLM call

Multi-Turn Context Enrichment

Short follow-up replies ("yes", "tell me more", "what about security?") have no semantic signal for the retriever. When a message is ≤4 words in an active conversation, the orchestrator enriches the retrieval query: 

retrievalQuery = "<last user question> <current short reply>"

This ensures embedding search targets the right topic even when the user's actual message contains no retrievable content on its own. The full conversation history is still passed to the Gemini API as a multi-turn contents array.

Three-Layer Guardrails

1
Security Block

Hardcoded blocklist of harmful terms (sql injection, xss, malware, exploit, ddos). Returns a refusal immediately — no LLM call.

2
Off-Scope Block

Explicit out-of-domain terms (write code, recipe, crypto trading, movie recommendation). Returns a scope explanation.

3
Domain-Ratio Check

For new conversations, checks the fraction of tokens matching GitLab domain terms (gitlab, handbook, ci, okr, pipeline…). Below 6% → likely off-topic. Skipped for active conversations to avoid blocking short follow-ups.

AI Critic with Automatic Retry

After the LLM generates an answer, the critic measures lexical overlap between answer tokens and retrieved source snippets. If fewer than 8% of answer tokens appear in the source material, the answer is flagged as weakly grounded and the orchestrator makes one more LLM call with an explicit instruction to "answer strictly from the provided context." This catches obvious cases of the model supplementing retrieved context with parametric knowledge.

Why lexical instead of semantic critic?

A semantic critic (embedding similarity) would be more accurate but costs one extra embedding API call per request. Lexical overlap runs in <1ms with no API dependency and catches the most egregious grounding failures.

Section 06

Key Trade-Offs

Next.js over FastAPI + Vite (separate services)

Benefit: Single git push → single Vercel deployment. No CORS, no two-service orchestration, no separate backend URL in frontend config.
Trade-off: Python evaluation and index-building scripts still require a local Python environment. These are offline dev-time tools, not runtime dependencies.

Commit vector_index.json to git (44MB)

Benefit: Vercel serverless functions have no persistent disk. Committing the index means it's available at runtime with no external vector DB.
Alternative rejected: Pinecone / Supabase pgvector — adds a paid dependency and cold-start latency.
Mitigation: Module-level caching in Node.js keeps the index in memory across warm requests; cold start pays the parse cost once.

Groq fallback (Llama 3.1 8B) for follow-up questions

Benefit: Follow-up generation is the lowest-stakes LLM call. If Gemini is rate-limited, Groq provides a free, fast alternative using an OpenAI-compatible API.
Why not for main answers: Llama 3.1 8B has lower factual accuracy and hallucination resistance than Gemini 2.0 Flash for domain-specific RAG.

Domain-ratio guardrail skipped for active conversations

Rationale: A short follow-up in an active conversation ("what about engineering?") will have near-zero domain-term ratio. Applying the check would block legitimate follow-ups. Skipping it for non-empty history is a deliberate UX decision.

Section 07

Future Improvements

01
Cross-encoder Re-ranking

Add a re-ranking step after initial vector retrieval to improve precision of the top-K sources before LLM generation. This would reduce hallucination risk on ambiguous queries.

02
Multi-hop Graph Traversal

Current knowledge graph retrieval uses one-hop entity lookup. Multi-hop traversal would improve answers to complex relational queries spanning multiple handbook sections.

03
Human Feedback Loop

Thumbs up/down feedback is already stored. Build a pipeline to surface systematically low-rated answers and use them to identify weak spots in retrieval or prompting.

04
Query Embedding Cache

The embedding API call is the main latency contributor (non-streamable). Caching recent query embeddings in KV would eliminate repeat-question latency significantly.

05
Semantic Critic

Replace lexical overlap with cosine similarity between the answer embedding and source embeddings for a more robust grounding signal — with rate-limit awareness to avoid added cost on every request.

06
Scheduled Index Sync

The incremental sync script exists but runs manually. A weekly GitHub Actions job running sync_index.py would keep the knowledge base current as the handbook updates.

Local Setup

Prerequisites

  • Node.js 20+
  • Gemini API key — aistudio.google.com/app/apikey (free tier works)

Run Locally

cd nextapp
cp .env.example .env.local
# Add GEMINI_API_KEY to .env.local

npm install
npm run dev
# Open http://localhost:3000

Without KV_REST_API_URL set, the app uses in-memory storage — fine for local testing. Data resets on server restart.

Deploy to Vercel

1
Import Repo

New Project → Import GitHub repo → Set Root Directory to nextapp/

2
Add Env Vars

Set GEMINI_API_KEY, JWT_SECRET, KV_REST_API_URL, KV_REST_API_TOKEN in Project Settings

3
Deploy

Click Deploy — every subsequent push to main triggers an automatic redeploy
